1. The delivery model

Design builds the full citizen-facing user experience in code, in the production stack, using the production design system. Back-end teams wire that front-end to real CMS and Azure services. The intent is to 10x the combined design and development timeline by eliminating the design→development handoff and removing UI re-implementation from the back-end critical path.

This inverts the traditional sequence. Instead of design producing mockups, back-end implementing those mockups in code, and then both iterating on what doesn't quite work — the design team ships a working Angular application that uses real components, real layouts, real interaction patterns, and mocked data. The back-end team picks up that working application and replaces the mocks with real APIs.

For this to work, two things have to hold:

  • Design works in the production stack. Same Angular, same @abgov/angular-components, same TypeScript that production will use. No re-implementation between design and prod.
  • The front-end shell is comprehensive. Design ships all in-scope needs as designed-in-code surfaces — not just the MVP wiring scope. Phase 2 surfaces ship as fully-designed-but-not-wired (or with explicit "coming soon" framing where they appear). This keeps the design work whole and de-risks phase 2 wiring.

2. The stack

Front-end (what design builds)

  • Angular
  • @abgov/angular-components (GoA Design System)
  • TypeScript
  • Mock data services in TypeScript interfaces — become API contracts at handoff
  • Deployable as a static Angular build (Azure Static Web Apps is the natural fit) for review and usability testing on real URLs

Back-end (what dev wires)

  • Node.js
  • Azure Functions for serverless processing (including notification workers)
  • Azure Web Services for hosted services
  • Azure SQL for the data layer
  • Implements the TypeScript API contracts the design build defines

Notification infrastructure (SMS + email) sits naturally in Azure Functions. The delegated-authority shared service (see §7) is a separate Azure service consumed by Intake Management, Case Management, and this portal — not part of the portal back-end specifically.

6. Two principles that shape sequencing

Coherent vertical slices, not back-end-easy cuts

If a stage is in MVP, all of its needs get wired end-to-end. "Show next payment but not payment history" or "Submit a change but not see the impact" creates exactly the disjointed experience that breaks user trust. The unit of MVP delivery is a complete stage, not an individual need.

Design covers everything, MVP is a wiring choice

Phase 2 needs are designed now, alongside MVP needs, in the same Angular build. They ship as completed designs awaiting back-end wiring. The cost of designing them now is small (same team, same components); the cost of designing them later is much larger.

7. Dependencies and risks

Biggest speed risk: delegated authority shared service

The delegated-authority capability — letting trustees, financial administrators, attorneys under POA, and other authorized representatives act on a recipient's behalf — must be built as a shared service consumed by Intake Management, Case Management, and the portal. Three reasons it's the dominant timeline risk:

  • It requires coordination across multiple consuming systems (not just portal work).
  • It's security/privacy-sensitive — auth and authorization for "on behalf of" relationships needs careful review, audit, and likely a privacy impact assessment.
  • It requires policy alignment on who can delegate to whom, under what evidence, with what revocation rules.

Other dependencies worth flagging

DependencyRisk levelNotes
Notifications infrastructure (SMS + email) Medium Azure Functions fit. Requires CMS event hooks for what triggers what. Notification preferences feature rides on top.
CMS read APIs Low–Medium Most MVP needs are reading data CMS already owns. Depends on existing API surface and how much new work is needed to expose what the portal needs.
CMS write APIs Medium Needed for any portal-initiated change (address updates, change reporting, document submission, benefit requests). Newer work, more careful auth scoping.
Re-application flow in Intake Management Low Confirmed: the existing application flow is leveraged. No new build in Intake Management to support Stage 9.
Alberta.ca account integration Low Auth only. No "on behalf of" — see delegated authority dependency above.
Federal-benefit data integration N/A Real federal integration is not feasible. The portal surfaces awareness and a recipient-side confirmation loop — no GoC data fetch needed.

8. Open questions to resolve early

None of these have to be answered today, but each one will silently slow the work if left open.

  1. Delegated-authority shared service — timeline and ownership. Who owns the spec? Who builds it? Is there appetite for the shared-service work, or is it likely to land on the portal team by default? When is it realistically usable?
  2. Where does the design build live during the design phase? Azure Static Web Apps is the natural fit, but choosing the deployment target affects whether real-user usability testing can happen on real URLs or stays local.
  3. API contract format. OpenAPI is the obvious choice for the handoff between the design build's TypeScript interfaces and the Azure Functions implementation. Worth confirming Case Management's existing conventions before the design build defines new contracts.
  4. Team composition. Specifically: are there Angular-comfortable designers, or does "design builds in code" require bringing in front-end developers who report to design rather than to the dev team? The model only works if the design team can actually ship code.
